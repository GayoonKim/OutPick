# Liked Tab Decision Details

## D-001: 좋아요 탭 이름은 `Liked`로 일반화한다

선택:

- 브랜드 전용 이름인 `LikedBrands`가 아니라 `LikedView`/`LikedViewModel`을 사용한다.

이유:

- 좋아요 탭은 장기적으로 브랜드 외에 시즌과 포스트를 함께 보여주는 보관함 화면이다.

트레이드오프:

- 리네이밍으로 파일 이동과 테스트명 변경이 발생한다.
- 대신 이후 시즌/포스트 확장 시 이름 충돌과 책임 혼란이 줄어든다.

## D-002: 좋아요 탭은 섹션별 상태로 분리한다

선택:

- 브랜드, 시즌, 포스트 섹션을 독립 상태로 관리한다.

이유:

- 한 섹션 실패가 다른 섹션 성공을 막으면 보관함 화면의 신뢰성이 떨어진다.
- 포스트 섹션 추가 시 독립 로딩/빈 상태/실패 상태가 필요하다.

보류한 대안:

- 좋아요 탭 전체를 하나의 전역 phase로 계속 관리하는 방식은 부분 실패 처리가 어렵다.

## D-003: 시즌 좋아요 목록은 user state 기준으로 조회하고 최신 Season 문서를 합성한다

선택:

- `users/{uid}/seasonStates`를 페이지 조회하고, 표시에는 최신 `Season` 문서를 다시 조회한다.

이유:

- 브랜드 좋아요 목록과 같은 패턴을 유지해 구현 일관성을 확보한다.
- 사용자별 좋아요 상태와 표시용 시즌 문서를 분리할 수 있다.

트레이드오프:

- 상태 문서 개수만큼 시즌 문서를 조회하므로 좋아요 수가 많아지면 성능 재검토가 필요하다.

재검토 조건:

- 좋아요 목록 로드가 느려지면 UseCase 내부 문서 재조회 병렬화 또는 제한된 batch 전략을 검토한다.

## D-004: 시즌 좋아요 변경은 Repository/UseCase 경계를 통해 처리한다

선택:

- View/ViewModel이 Cloud Functions를 직접 알지 않고 Repository/UseCase를 통해 처리한다.

이유:

- 기존 MVVM-C + Repository + UseCase + DI 흐름을 유지한다.

## D-005: 중복 로드 방지는 `@MainActor struct AsyncLoadGate`로 정리한다

선택:

- 화면 private 로딩 gate는 `actor`가 아니라 `@MainActor struct`로 둔다.

이유:

- `LikedViewModel`은 `@MainActor`이고 gate도 화면 private 상태다.
- actor를 쓰면 불필요한 `await`와 lifecycle 복잡도가 늘어난다.

보류한 대안:

- `actor AsyncLoadGate`는 여러 객체가 공유하거나 MainActor 밖에서 접근하는 전역 작업 큐/레지스트리에는 적합하지만, 현재 화면 private gate에는 과하다.

재검토 조건:

- 업로드 큐, 전역 task registry, 여러 화면 공유 작업 상태가 생기면 actor 기반 service를 별도로 설계한다.

## D-006: 원격에만 남은 `updateBrandLogoDetailPath` 함수는 삭제하지 않는다

선택:

- Functions 전체 배포 중 삭제 확인이 필요했던 원격 함수는 사용자 명시 삭제 요청 없이 삭제하지 않는다.

이유:

- 운영 영향이 불명확한 원격 함수 삭제는 되돌리기 어렵다.

재검토 조건:

- `updateBrandLogoDetailPath(asia-northeast3)`가 더 이상 필요 없는지 확인되면 별도 삭제 배포를 검토한다.

## D-007: 좋아요 포스트 목록은 2열 grid와 기존 PostDetail push를 사용한다

선택:

- 좋아요 포스트 섹션은 2열 `LazyVGrid`로 표시한다.
- 포스트 셀 탭은 기존 `PostDetailView`로 push한다.
- 포스트 카드에는 이미지 중심 정보를 보여주고 브랜드/시즌 텍스트 정보는 기본 노출하지 않는다.

이유:

- 포스트 좋아요는 사용자가 마음에 든 사진/스타일을 다시 보는 목적이 크다.
- 기존 PostDetail 흐름을 재사용하면 댓글, 좋아요, 저장, 상세 미디어 상태와 navigation 책임이 한 곳에 유지된다.

트레이드오프:

- grid 카드 자체의 정보량은 낮다.

## D-008: 좋아요 취소는 카드 메뉴에서 optimistic remove 후 실패 시 복구한다

선택:

- 브랜드/시즌/포스트 카드 오른쪽 상단에 점 3개 메뉴를 두고 `좋아요 취소` 액션을 제공한다.
- 액션 실행 시 로컬 목록에서 먼저 제거하고 서버 반영에 실패하면 기존 위치로 복구한다.

이유:

- 좋아요 탭의 주 액션은 콘텐츠 열람이고, 취소는 보조 액션이라 카드 전체 탭과 분리하는 편이 안전하다.
- 즉시 제거는 사용자가 취소 결과를 바로 이해하기 쉽다.

트레이드오프:

- 실패 복구 시 사용자가 제거됐던 카드가 다시 나타나는 순간이 생길 수 있다.

## D-009: 좋아요 탭 진입은 앱 실행 중 캐시를 우선하고 pull-to-refresh로 서버 최신화한다

선택:

- 좋아요 탭 최초 로드 후 앱 실행 중 재진입은 로컬 store/cache 상태를 우선 적용한다.
- 서버 강제 최신화는 pull-to-refresh 또는 앱 재시작/초기 로드 흐름에 둔다.
- 브랜드/시즌/포스트 좋아요 변경은 interaction store invalidation으로 로컬 즉시 반영한다.

이유:

- 좋아요 탭은 기본적으로 해당 기기 사용자의 보관함이며, 사용자가 앱 실행 중 직접 만든 변경의 즉시 반영이 더 중요하다.

트레이드오프:

- 다중 기기에서 발생한 외부 변경은 pull-to-refresh 전까지 바로 반영되지 않을 수 있다.

재검토 조건:

- 외부 변경 즉시 동기화가 제품 요구사항이 되면 listener 또는 refresh 정책을 재검토한다.
