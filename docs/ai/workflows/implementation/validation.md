# Implementation Validation

## 목적

구현 전 가치 기준, 작업별 역할 관점, 테스트/QA 경계, 미배포 앱 호환성 기준을 정리한다.

## 미배포 앱 호환성 기준

- OutPick 앱은 아직 외부 배포 이력이 없으므로, 배포된 앱 버전이나 실제 사용자의 기존 클라이언트와의 하위 호환성을 기본 제약으로 두지 않는다.
- 상태 enum, DTO, API payload, Firestore 필드, 화면 흐름을 더 명확하게 정리할 수 있다면 사용되지 않는 기존 형식을 병행 지원하지 않고 제거하는 방향을 우선한다.
- 테스트 fixture, 운영 smoke 데이터, 개발 중 생성된 문서는 실제 사용자 호환성 근거로 보지 않는다.
- 호환 계층을 추가하려면 실제 배포 이력, 보존해야 할 운영 데이터, 외부 소비자, 단계적 rollout 필요성 중 하나 이상의 구체적 근거가 있어야 한다.
- 앱이 TestFlight, App Store, 사내 배포 등으로 실제 사용자에게 제공되기 시작하면 이 원칙을 재검토하고 배포 버전별 호환성 정책을 별도 ADR로 정한다.
- 데이터 삭제나 운영 데이터 마이그레이션은 앱 미배포 여부와 별개로 되돌리기 어려운 작업이므로 사용자 명시 승인 없이 실행하지 않는다.

## 가치 기준

기술 선택 전 아래 항목을 짧게 정리한다.

- 역할 관점: 이번 작업의 핵심 가치 제공을 가장 잘 검증할 수 있는 기업, 제품, 도메인, 전문 역할 관점을 정한다. 특정 회사를 기본값으로 삼지 않고, 작업 성격에 맞게 예를 들어 운영 큐/대규모 백엔드는 Google/AWS/Netflix 계열 시니어 백엔드/CTO 관점, 커머스 상품 탐색은 Shopify/Farfetch 계열 시니어 PM 관점, 앱 사용성은 iOS HIG에 익숙한 시니어 iOS/디자인 리드 관점, 회귀 검증은 시니어 QA 관점, 정책/심사는 App Store 심사 경험자 관점처럼 선택한다.
- 서비스 핵심 가치: OutPick이 사용자에게 주는 가장 중요한 가치.
- 기능 핵심 가치: 이번 기능이 서비스 핵심 가치에 기여하는 방식.
- 품질 속성 우선순위: 속도, 정확도, 안정성, 확장성, 비용, 운영 단순성 중 무엇이 더 중요한지.
- 선택한 기술과 대안: 선택지별 장단점과 현재 phase에서 감수하는 트레이드오프.
- 재검토 조건: 어떤 지표나 사용자 피드백이 나오면 결정을 바꿀지.

이 기준이 모호하면 구현을 시작하지 않고 사용자와 논의한다.
역할 관점은 권위에 기대어 단정하기 위한 장치가 아니라, 어떤 리스크와 성공 기준을 더 엄격하게 볼지 정하는 사고 프레임으로 사용한다.

## 테스트/QA 경계

자동 테스트는 모든 동작을 코드로 고정하기 위한 장치가 아니라, 사람이 안정적으로 재현하기 어렵거나 회귀 비용이 큰 부분을 보호하기 위한 장치다.

자동 테스트 우선 후보:

- 서버 실패, 빈 응답, 권한 실패, 일부 API만 실패처럼 실제 앱 실행으로 제어하기 어려운 케이스.
- 비동기, 중복 호출 방지, 캐시, pagination, race condition, Publisher/Combine 상태 발행.
- Repository, UseCase, ViewModel의 상태 전이와 도메인 규칙.
- 로그인 유저 없음, 좋아요 취소 후 목록 제거, 부분 실패 시 다른 섹션 유지처럼 회귀 위험이 큰 분기.

수동 QA 우선 후보:

- 시뮬레이터나 실기기에서 짧게 확인 가능한 단순 happy path UI.
- 화면의 시각적 완성도, 터치 흐름, 스크롤, navigation 감각.
- 실제 Firebase 데이터가 화면에 표시되는지 확인하는 통합 수준의 앱 실행 점검.

원칙:

- 테스트나 QA를 실행하기 전에는 어떤 검증을 왜 하는지, 어떤 실패를 잡으려는지, 성공/중단 기준이 무엇인지 먼저 정리하고 사용자와 논의한다.
- 실제 서버 상태를 사용자가 제어하기 어려운 케이스는 fake repository, fake use case, spy로 자동 테스트한다.
- 수동 QA로 충분한 항목은 테스트 코드를 만들지 않은 이유와 확인할 체크리스트를 `progress.md`에 남긴다.
- 자동 테스트와 수동 QA 중 무엇이 맞는지 애매하면 구현 전에 사용자와 논의한다.

검증 전 논의 형식:

```text
검증 전 점검:
- 역할 관점:
- 검증 목표:
- 실행할 테스트/QA:
- 이 검증이 필요한 이유:
- 성공 기준:
- 실패 또는 중단 기준:
- 운영/데이터/비용 리스크:
- 사용자 확인 필요 항목:
```
