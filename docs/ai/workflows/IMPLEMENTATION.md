# Implementation Workflow

## 목적

설계가 끝난 기능을 phase 단위로 구현하고, 진행상황과 반복 지식을 하네스에 반영하기 위한 워크플로우다.

## 순서

1. `docs/ai/ENTRYPOINTS.md`에서 관련 진입점을 확인한다.
2. 관련 task 문서가 있으면 `docs/ai/tasks/{task-name}`을 확인한다.
3. 하네스 문서에 정보가 부족하면 필요한 코드 범위만 탐색한다.
4. 구현 중 제품 또는 기술 결정이 모호하면 사용자와 논의한다.
5. phase 단위 구현 계획을 작성한다.
6. 사용자 승인 후 파일을 수정한다.
7. 변경 범위에 맞는 검증 방법을 정한다.
8. 검증을 수행했으면 결과를 기록하고, 수행하지 않았으면 이유를 기록한다.
9. 반복 재사용될 구조, 진입점, 검증 명령, 기술 결정은 `docs/ai` 갱신 후보로 제안한다.
10. 같은 명령이나 검증 흐름이 2~3회 이상 반복되면 `scripts/ai` 자동화 후보로 제안한다.

## 논의 필요 기준

아래 항목이 모호하면 구현을 멈추고 사용자와 논의한다.

- 기존 설계 문서와 실제 코드가 충돌하는 경우
- 변경 범위가 요청보다 커지는 경우
- 화면 이동 책임을 View, ViewModel, Coordinator 중 어디에 둘지 애매한 경우
- Repository, UseCase, Store, Cloud Functions 중 어느 계층에 로직을 둘지 애매한 경우
- 데이터 삭제, 마이그레이션, 보안 규칙 완화, 운영 배포처럼 되돌리기 어려운 작업
- 테스트 실행 여부가 비용과 위험 사이에서 애매한 경우
- 하네스 문서에 없는 구조를 새로 발견했고 앞으로 반복 사용될 가능성이 있는 경우

## 논의 방식

논의가 필요한 경우 아래 형식으로 보고한다.

```text
논의 필요:
- 모호한 지점:
- 가능한 선택지:
- 각 선택지의 장단점:
- 추천안:
- 사용자 결정이 필요한 항목:
```

사용자 결정 전에는 해당 결정에 의존하는 코드 수정, 배포, 삭제, 마이그레이션을 진행하지 않는다.

## 작업 문서 기준

- `plan.md`: phase 단위 구현 계획.
- `progress.md`: 현재 진행상황, 완료한 일, 남은 일, 변경 파일, 검증 상태.
- `decisions.md`: 해당 작업 안에서 생긴 결정. 프로젝트 전체에 영향이 있으면 `docs/ai/ADR.md`로 승격한다.

## scripts/ai 자동화 기준

`scripts/ai`는 반복이 확인된 작업만 만든다.

추가 후보:

- 같은 `xcodebuild` 빌드/테스트 명령을 2~3회 이상 반복했다.
- 같은 Firebase Functions lint/build 명령을 2~3회 이상 반복했다.
- 같은 Firestore rules/indexes 검증 또는 배포 전 점검을 2~3회 이상 반복했다.
- 커밋 전 변경 파일 분류를 매번 수동으로 반복하고 있다.
- 공식/로컬 하네스 분리 상태 확인을 반복하고 있다.

추가하지 않을 것:

- 한 번만 필요한 임시 명령.
- 아직 명령 인자나 실행 조건이 안정되지 않은 작업.
- 사용자 승인 없이 배포, 삭제, 마이그레이션을 실행하는 스크립트.
- 프로젝트 이해보다 자동화 자체가 더 복잡해지는 스크립트.

제안 형식:

```text
scripts/ai 자동화 후보:
- 반복된 명령/흐름:
- 반복 횟수:
- 스크립트로 만들면 줄어드는 비용:
- 위험 또는 승인 필요 여부:
```
