# New Feature Workflow

## 목적

새 기능 또는 큰 수정 요청이 들어왔을 때 바로 구현하지 않고 설계 하네스를 먼저 진행하기 위한 워크플로우다.

## 순서

1. 기존 문서 확인
   - `docs/ai/PRD.md`
   - `docs/ai/FLOW.md`
   - `docs/ai/SCREEN_SPEC.md`
   - `docs/ai/DATA_SCHEMA.md`
   - `docs/ai/CODE_ARCHITECTURE.md`
   - `docs/ai/ENTRYPOINTS.md`
   - `docs/ai/ADR.md`
2. 요구사항 정리
3. 구현 디테일 정리
4. 제약 조건 정리
5. 완료 기준 정리
6. 구현 가능성 검증
7. 기술 스택 선정 또는 기존 스택 적합성 확인
8. 사용자 흐름 점검
9. 화면 설계
10. API 설계 필요 여부 확인
11. 데이터 설계
12. 코드 아키텍처 영향 확인
13. 기술적 결정사항 점검
14. 최종 문서 생성 또는 갱신

## 논의 필요 기준

아래 항목이 모호하면 구현하지 않고 사용자와 논의한다.

- 요구사항
- 기능 범위
- 구현 디테일
- 제약 조건
- 완료 기준
- 사용자 흐름
- 화면 수와 화면 이동
- 데이터 저장 위치
- API 설계와 호출 주체
- 서버 또는 Firebase Functions 필요 여부
- 테스트/검증 기준
- 자동 테스트와 수동 QA의 경계
- 권한, 개인정보, 결제, UGC, AI, 외부 링크 등 정책 리스크
- 기존 아키텍처와 충돌 가능성
- 사용자 경험에 큰 영향을 주는 결정
- 되돌리기 어려운 배포, 삭제, 마이그레이션

## 논의 방식

논의가 필요한 경우 아래 형식으로 사용자에게 묻는다.

```text
논의 필요:
- 모호한 지점:
- 가능한 선택지:
- 각 선택지의 장단점:
- 추천안:
- 사용자 결정이 필요한 항목:
```

사용자 결정 전에는 문서 확정, task 생성, phase 생성, 코드 수정을 진행하지 않는다.

## 구현 전 조건

- 사용자가 구현 진행을 승인해야 한다.
- task/phase 계획에는 목표, 변경 범위, 완료 기준, 검증 방법, 논의 필요 사항을 포함한다.
