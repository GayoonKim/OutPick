# ADR-019: 핵심 인프라는 기능별 모듈러 경계와 현재 배포 단위를 유지한다

상태: accepted

결정:

- CloudFunctionsManager.swift, GRDBManager.swift, functions/src/index.ts, Socket/index.js에 집중된 책임을 기능별 모듈로 분리한다.
- 단순 file extension 분리가 아니라 소비자 capability별 Protocol, 기능별 Client/Store/handler/service, 공통 transport/database/runtime, 얇은 entrypoint를 사용한다.
- iOS View/ViewModel/UseCase/Repository 흐름은 필요한 좁은 계약에 의존하며 FirebaseFunctions SDK와 GRDB concrete 구현을 직접 알지 않는다.
- iOS 공통 CloudFunctionsTransport는 SDK 호출, region, 공통 오류 같은 횡단 책임만 가진다.
- iOS 공통 AppDatabase는 DatabasePool 생성과 migration 실행을 소유하고 기능별 GRDB Store에 주입한다.
- Firebase Functions index.ts는 기존 Function 이름의 flat export 진입점으로 유지한다.
- Socket index.js는 server와 dependency 생성, middleware/handler 등록, start/shutdown 조립만 담당한다.
- 현재 배포 경계인 OutPick iOS 앱, Firebase Functions default codebase, outpick-socket Cloud Run service를 유지한다.
- 이번 결정만으로 Firebase codebase 분리, 새 Cloud Run microservice, Kubernetes를 도입하지 않는다.

이유:

- 여러 개발자가 서로 다른 기능을 담당할 때 giant concrete type과 대형 entrypoint는 소유권, 코드 리뷰, merge conflict, 회귀 검증 범위를 불필요하게 넓힌다.
- 파일만 분리하고 같은 giant manager 전체를 주입하면 소비자 의존성과 테스트 비용은 줄지 않는다.
- capability Protocol은 소비자가 실제로 필요한 계약을 명시하고 fake/spy로 대체할 수 있게 한다.
- 공통 runtime 생성 책임을 한 곳에 두면 SDK 초기화와 connection/pool이 기능 파일마다 중복되는 것을 막을 수 있다.
- 현재 문제는 source/dependency 집중이며 독립 서비스가 필요한 scale, IAM, 장애, 데이터 소유권 근거는 아직 없다.
- OutPick은 이미 iOS 앱, Functions, Socket Cloud Run이라는 서로 다른 runtime 경계를 가지고 있다.

트레이드오프:

- source file, Protocol, concrete type, DI 조립 코드가 늘어난다.
- 기존 singleton 직접 호출을 생성자 주입으로 전환하는 초기 변경량이 크다.
- capability를 지나치게 작게 나누면 type과 naming 비용이 증가할 수 있다.
- 같은 배포 단위 안에 있으므로 한 기능 변경이 같은 앱 또는 service revision 배포를 요구할 수 있다.
- modular monolith는 process-level 장애 격리와 독립 autoscaling을 제공하지 않는다.

보류한 대안:

- 같은 manager를 여러 extension 파일로만 나누는 방식은 탐색성은 좋아지지만 dependency boundary가 개선되지 않아 보류한다.
- 네 영역을 한 번에 big-bang 전환하는 방식은 wire/data/transaction 회귀 원인과 rollback 단위가 커져 보류한다.
- 기능마다 Firebase codebase 또는 Cloud Run service를 만드는 MSA는 네트워크 계약, 인증, 관측성, 분산 실패와 운영 비용이 현재 이점보다 커서 보류한다.
- Kubernetes는 현재 Cloud Run/Firebase 운영 요구를 해결하는 데 필요하지 않아 보류한다.

재검토 조건:

- 기능별 독립 배포 요구가 반복되고 같은 codebase 배포가 실제 병목이 된다.
- 특정 기능만 독립 autoscaling, timeout, region, IAM 또는 비용 경계가 필요하다.
- 장애 전파를 process 수준에서 격리해야 한다.
- 팀과 데이터 소유권이 기능별로 장기간 독립되고 서비스 간 계약 운영 역량이 준비된다.
- modular monolith 내부의 의존 규칙만으로 변경 충돌과 결합도를 제어하기 어려운 증거가 생긴다.

구현 원칙:

- 함수 하나당 Protocol 하나를 만들지 않고 같은 소비자와 변경 이유를 공유하는 capability로 묶는다.
- 공통 계층에는 기능별 payload, domain 정책, query orchestration을 넣지 않는다.
- GRDB의 여러 table을 함께 변경하는 operation은 transaction owner를 한 곳에 둔다.
- Functions export 이름과 Socket event/ACK 같은 외부 계약 변경은 구조 리팩터링과 분리해 별도 승인한다.
- 운영 배포는 관련 workflow와 사용자 명시 승인을 따른다.
