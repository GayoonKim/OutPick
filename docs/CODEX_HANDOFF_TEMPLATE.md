# Codex Handoff Template

긴 리팩토링, context 압축 전, 세션 전환 전, 큰 단계 완료 직후에 `HANDOFF.md`를 아래 형식으로 갱신한다.

불확실한 내용은 임의로 단정하지 않고 `재확인 필요`로 표시한다. 추론으로 작성한 내용은 `추측입니다`라고 표시한다.

## 1. 최종 목표

- 이번 작업의 최종 목표를 한두 문단으로 적는다.
- 해결하려는 핵심 문제와 성공 기준을 함께 적는다.

## 2. 완료한 작업

- 이미 구현하거나 검증한 작업만 적는다.
- 코드 변경, 문서 변경, 빌드/테스트 결과를 분리해서 적는다.
- 완료 여부가 애매하면 완료로 적지 말고 `재확인 필요`로 표시한다.

## 3. 아직 남은 작업

- 다음에 실제로 해야 할 작업을 우선순위대로 적는다.
- 보류한 이유가 있으면 함께 적는다.

## 4. 수정한 파일 목록

- 수정/추가/삭제한 파일을 경로 기준으로 적는다.
- 파일별로 핵심 변경 내용을 짧게 적는다.
- working tree 상태를 확인하지 못했으면 `재확인 필요`로 표시한다.

## 5. 중요한 아키텍처 결정

- 어떤 구조를 선택했는지 적는다.
- 왜 그 구조를 선택했는지 적는다.
- 기존 MVVM-C, Repository, UseCase, DI, Coordinator 흐름과의 관계를 적는다.
- 복잡한 요구사항 속에서 단순하고 유지 가능한 구조를 만들기 위해 어떤 균형점을 선택했는지 적는다.
- 속도와 품질 사이에서 현실적으로 선택한 트레이드오프를 적는다.
- 선택하지 않은 대안과 그 이유를 적는다.
- 나중에 더 높은 품질의 구조로 확장하거나 재검토해야 하는 조건을 적는다.

예시:

```text
선택:
- 단일 post 화면은 key 단위 publisher로 전환했지만, 리스트 화면은 기존 dictionary publisher를 유지했다.

이유:
- 단일 화면의 불필요한 이벤트 반응은 줄이되, 리스트 화면까지 한 번에 바꾸면 변경 범위가 커지고 검증 비용이 증가한다.

트레이드오프:
- 단기적으로 facade publisher가 일부 남지만, 현재 화면별 요구에는 충분히 단순하고 안전하다.

보류한 대안:
- 모든 화면을 Store별 key publisher로 전면 전환하는 방식은 이상적이지만, 이번 단계에서는 리스크와 작업량이 크다.

재검토 조건:
- 리스트 post 수가 크게 늘거나 interaction 이벤트가 빈번해져 렌더링 비용이 실제 병목으로 확인될 때 재검토한다.
```

## 6. 다시 확인해야 할 불확실한 부분

- 실제 코드를 다시 읽어야 하는 부분은 `재확인 필요`로 표시한다.
- 추측으로 보이는 내용은 `추측입니다`라고 표시한다.
- 정책, 외부 문서, 최신 정보가 필요한 내용은 공식 문서 확인 필요 여부를 적는다.

## 7. 다음 턴에서 바로 실행해야 할 작업

- 다음 assistant turn에서 첫 번째로 해야 할 일을 적는다.
- 가능하면 실행 명령, 확인할 파일, 수정 방향까지 구체적으로 적는다.

## 권장 요청 문구

```text
현재 작업 상태를 HANDOFF.md에 7개 항목으로 갱신해줘.
추측은 추측이라고 표시하고, 파일을 다시 읽어야 하는 부분은 재확인 필요라고 표시해.
```

```text
압축 전에 현재 작업 상태를 7개 항목으로 정리해줘.
완료한 작업과 남은 작업을 구분하고, 불확실한 부분은 임의로 추측하지 말고 재확인 필요라고 표시해.
```

```text
HANDOFF.md를 기준으로 현재 목표, 완료한 일, 남은 일, 수정 파일을 다시 말해줘.
요약과 실제 파일 상태가 다르면 재확인 필요라고 표시해.
```
